How the CSS cascade actually resolves a color

You wrote color: teal in six places. The pixel is one color. Here is the exact machinery — origin, specificity, order, inheritance, and computed values — that picks the winner.

Mara Delacroix • July 22, 2026 • 11 min read

Ask ten engineers why a link came out teal instead of blue, and you will get ten confident, partly-wrong answers. "Specificity won." "It was last in the file." "Inline styles always win." Each is a fragment of a single, fully-specified algorithm that the browser runs millions of times per page — and once you can run that algorithm in your head, CSS stops feeling like a slot machine.

The cascade is the part of CSS that decides, for one element and one property, which declared value survives. It does not lay anything out and it does not paint anything. It answers exactly one question — whose value wins? — and it answers it deterministically. The confusion is almost never about the rules; it is about the order in which the rules apply and where each one stops mattering.

The four questions, asked in order

When more than one declaration targets the same element and property, the browser sorts every candidate by four criteria, checked strictly top to bottom. It only moves to the next criterion when the current one is a tie. That ordering is the whole game:

  • Origin and importance. Where did the rule come from — the browser's default sheet, the user's stylesheet, or the author's — and was it marked !important? Author-important beats author-normal beats user-normal beats the user agent.
  • Specificity. Only among declarations that survived step one. An ID selector outranks a class, which outranks a type selector. The * universal selector contributes nothing.
  • Source order. Still tied? The declaration that appears later in the resolved stylesheet order wins. This is the only place "last one wins" is actually true.
  • Inheritance and defaults. If nothing was declared, the property either inherits from the parent's computed value or falls back to its initial value.

Notice what is not on that list. Proximity in the DOM does not matter. How "recently" you wrote the rule does not matter. Whether the selector "looks more specific" to a human does not matter. The engine compares structured tuples, and structured tuples do not have opinions.

Specificity is not a score you accumulate — it is a three-column number you compare left to right. A single ID beats a thousand classes, because the columns never carry. — the mental model that fixes 80% of cascade bugs

Specificity is a tuple, not a total

The most common misconception is that specificity is additive weight — that enough classes eventually "add up" to an ID. They do not. Specificity is the tuple (A, B, C): A counts ID selectors, B counts classes, attributes and pseudo-classes, and C counts type selectors and pseudo-elements. Comparison is lexicographic. (1,0,0) beats (0,13,0) every time, because you compare column A first and stop the moment one side is larger.

This is also why the newer :where() pseudo-class is such a useful tool: it wraps a selector list but contributes zero specificity, letting you write low-priority defaults that any later, more specific rule can trivially override. Its sibling :is(), by contrast, takes the specificity of its most specific argument.

Where !important actually sits

People treat !important as a bigger number bolted onto specificity. It is not — it lives one level up, in step one. Marking a declaration important moves it into a different, higher-priority origin bucket before specificity is ever consulted. That is why an important author declaration with specificity (0,0,1) beats a normal author declaration at (1,0,0): they are no longer being compared in the same round.

A useful inversion: the important origins run in reverse. Among !important declarations, the user agent's win over the user's, which win over the author's — the exact opposite of the normal order. That is the mechanism behind accessibility overrides: an important user rule for high-contrast text can defeat any author styling.

Cascade layers changed the shape of step two

For two decades the cascade had a stable silhouette, and then @layer quietly inserted a new tier between importance and specificity. Cascade layers let you declare an explicit priority order for whole groups of rules, so a low-layer rule loses to a high-layer rule regardless of specificity. A tiny type selector in your components layer can now beat a heavyweight ID selector sitting in your reset layer.

This is the single most important recent change to how the cascade resolves, because it moves the tie-break away from the specificity arms race that produced selectors like #app #main .card.card.card. With layers, you order intent explicitly instead of encoding it in selector length:

/* declare priority once, low → high */
@layer reset, base, components, utilities;

@layer reset {
  #hero a { color: blue; }   /* specificity (1,0,1) … */
}

@layer components {
  a { color: teal; }        /* … still loses to this (0,0,1) */
}
/* resolved: teal — a later layer outranks earlier layers */

The full priority ladder, from weakest to strongest, is now: user-agent styles, then the layered and unlayered normal declarations sorted by origin and layer order, then transitions, then the important declarations in reverse-origin order, and finally animations sit apart because active @keyframes override normal declarations while yielding to important ones. Most days you only touch the middle of that ladder — but knowing the ends exist is what lets you debug the weird cases.

From "specified" to "computed": the value keeps moving

Winning the cascade does not hand you a pixel color. The surviving declaration is the specified value. The browser then resolves it into a computed value — relative units become absolute-ish, currentColor and inherit get dereferenced, and custom properties are substituted. Only later, at used-value time, does color: teal become the concrete rgb(0 128 128) that the painter rasterizes.

This staging matters because inheritance operates on the computed value, not the specified one. When a child inherits color, it copies the parent's already-resolved value. So if a parent's color came from a custom property that later changed, the cascade you reasoned about upstream is exactly the input the child inherits downstream — no re-running of selectors, just a value handed down the tree.

The cascade picks a value once, per element, per property. Everything downstream — inheritance, computed values, the painted pixel — is bookkeeping on that decision.

So the next time a link renders teal when you swore it should be blue, do not reach for another !important. Walk the four questions in order: which origin bucket, which layer, which specificity tuple, which source position. The answer is always there, and it is always the same answer the engine found — because you are running the same algorithm it did.
